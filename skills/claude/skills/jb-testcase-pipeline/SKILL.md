# 经贝测试用例流水线 v2（JB Testcase Pipeline）

基于 `jb-pm-pipeline` 产出的 PRD 和 `jb-td-pipeline` 产出的技术设计文档，系统化生成测试用例说明书。

## 核心能力

1. **智能输入解析** — 自动检测前置文档，分级校验，增量检测
2. **功能类型检测** — 自动判断纯前端/前端+API/后端逻辑/全栈
3. **两阶段 Agent 生成** — Phase A 并行 BE/FE/API → Phase B 串行生成 BB（真实追溯链）
4. **代码上下文注入** — 按 PRD 关键词精准搜索代码，注入类名/方法/组件结构
5. **覆盖方向自适应** — 按功能复杂度选择覆盖方向，简单功能不凑数
6. **结构化覆盖率** — checklist 对比，非 Agent 自评
7. **增量生成** — 自动检测已有用例，只补充缺失功能点
8. **测试数据管理** — 自动生成测试数据建议

## 前置条件（分级）

### Minimal（必须）
- `prd.md` — 产品需求文档
- `design/backend/api.md` 或 `design/frontend/components.md`（至少一个）

### Full（推荐）
- `design/backend/architecture.md`
- `design/backend/database.md`
- `design/frontend/architecture.md`
- `design/frontend/api-integration.md`
- `plans/plan.md`

## 状态机

```
NO_STATE ──→ initialized ──→ generating ──→ generated ──→ coveragechecked ──→ completed
                │                              │               │
                ↓                              ↓               ↓
             failed                         failed          failed
```

| 状态 | 说明 | 处理 |
|------|------|------|
| `NO_STATE` | 首次使用 | 创建状态文件，进入 Step 0 |
| `initialized` | 输入校验完成 | 进入 Step 1 |
| `generating` | Agent 生成中 | 等待完成或重试 |
| `generated` | 用例已生成 | 进入覆盖率计算 |
| `coveragechecked` | 覆盖率确认完成 | 进入 Step 3/4 |
| `completed` | 已完成 | 查看/复用/新建 |
| `failed` | 失败 | 显示错误，允许重试 |

## 质量门禁（必须全部通过才能输出）

### Step 0 输入解析门禁

| # | 门禁 | 检查方式 | 不通过处理 |
|---|------|----------|------------|
| G1 | prd.md 存在且非空 | 读取文件，检查内容 | 停止，提示提供 PRD |
| G2 | 设计文档至少一个存在 | 检查 api.md 或 components.md | 停止，提示提供设计文档 |
| G3 | 功能类型已检测 | 分析 PRD 关键词 + 搜索代码 | 停止，必须先确定类型 |

### Step 1 用例生成门禁

| # | 门禁 | 检查方式 | 不通过处理 |
|---|------|----------|------------|
| G3.5 | **全栈需求双端代码已找到** | 功能类型为"全栈"时，必须同时找到前端项目和后端项目代码 | **只找到一端时，必须主动询问用户另一端代码位置，禁止使用泛化类名** |
| G4 | 后端代码已读取 | 搜索 `@TableName`、Service、Controller | 停止，必须理解后端逻辑 |
| G5 | 前端代码已读取 | 搜索页面组件、路由、Store | 停止，必须理解前端交互 |
| G6 | API 接口已确认 | 搜索接口定义、请求/响应结构 | 停止，必须理解数据流 |
| G7 | 数据库表名已验证 | `@TableName` 注解确认实际表名 | 停止，禁止猜测表名 |
| G8 | 按功能类型+复杂度决定生成层 | 纯前端→跳过 BE/API，后端逻辑→跳过 FE | 自动跳过不需要的层 |

### Step 2 覆盖率门禁

| # | 门禁 | 检查方式 | 不通过处理 |
|---|------|----------|------------|
| G9 | 需求覆盖率 ≥ 80% | PRD 功能点 vs 已覆盖功能点 | 自动补充缺失用例 |
| G10 | 覆盖率报告已输出 | 检查 coverage.json 是否生成 | 补充输出 |

| 覆盖率 | 判定 | 处理 |
|--------|------|------|
| ≥ 90% | 优秀 | 直接通过 |
| 80%-89% | 合格 | 通过，建议补充 |
| 60%-79% | 不合格 | 自动补充缺失用例 |
| < 60% | 严重不足 | 重新生成 |

### Step 3-4 输出门禁

| # | 门禁 | 检查方式 | 不通过处理 |
|---|------|----------|------------|
| G11 | cases.md 已生成 | 检查文件存在 | 停止，必须输出用例文档 |
| G12 | 用例编号连续 | 每层从 TC-XX-001 自增 | 重排编号 |
| G13 | 当前功能测试数据已就绪 | 检查 `{需求目录}/test/case/testdata.md` | 提示可后续补充 |

### Step 5 E2E 生成门禁（选择 E2E 时）

| # | 门禁 | 检查方式 | 不通过处理 |
|---|------|----------|------------|
| G14 | 前端页面组件已读取 | 搜索 `views/**/*.vue`，读取目标页面 | 停止，必须分析 DOM 结构 |
| G15 | 后端接口已读取 | 搜索 API 调用，理解请求/响应 | 停止，必须理解数据流 |
| G16 | 路由配置已确认 | 搜索路由文件，确认 hash 路径 | 停止，URL 写错测试必失败 |
| G17 | 已有 E2E 脚本已读取 | 读取 `tests/e2e/**/*.spec.ts` | 停止，必须遵循已有约定 |
| G18 | 每个 spec 有对应 Page Object | 检查 import 的 Page 是否存在 | 先生成 Page Object |
| G19 | 未实现功能用 test.skip() | 检查功能是否已实现 | 未实现必须 skip |
| G20 | 运行验证通过 | `npx playwright test --list` + 实际运行 | 修复或标记 skip |

## 执行入口

```bash
# 基本用法 - 使用迭代版本号（推荐）
/jb-testcase-pipeline v6.8.8

# 指定目录
/jb-testcase-pipeline /path/to/ai-doc/迭代/feat-1234-xxx

# 飞书编号
/jb-testcase-pipeline FEAT-1234

# 仅生成后端测试
/jb-testcase-pipeline v6.8.8 --scope backend

# 仅生成黑盒测试
/jb-testcase-pipeline v6.8.8 --scope blackbox

# 生成测试代码骨架
/jb-testcase-pipeline v6.8.8 --with-codegen
```

## 工作区检测

Skill 会自动检测工作区路径（包含 `ai-doc/迭代` 的目录）：

1. **自动检测**：扫描常见位置（D:/work, D:/workspace, E:/work 等）
2. **当前目录**：检查当前工作目录是否包含 `ai-doc/迭代`
3. **向上查找**：从当前目录向上查找
4. **手动输入**：如果未找到，询问用户提供工作区路径

检测到的工作区路径会保存在状态文件中，后续操作使用该路径。

## 参数说明

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `--scope` | 生成范围：all/backend/frontend/api/blackbox/e2e | all |
| `--level` | 输入级别：minimal/full | auto |
| `--incremental` | 增量模式：已有 cases.md 时只补充缺失功能点 | auto（自动检测） |
| `--with-codegen` | 是否生成测试代码骨架 | false |
| `--e2e-output` | E2E 测试脚本输出路径 | 自动搜索 + 用户选择 |
| `--max-tokens` | 单个 Agent 最大输出 token | 4000 |
