# Step 4: 测试用例同步

> **目标**：对比测试用例文档（test/case/cases.md）与实际代码和测试结果，更新测试用例状态。

## 检查

如果 `phases.testcaseSync == true` 且 `test/case/cases.md` 已更新过：

AskUserQuestion：
> 测试用例已同步过，复用还是重新对账？
> - **A)** 复用
> - **B)** 重新对账

## 执行

### 4.1 分析代码变更与测试覆盖

启动 Agent 分析测试用例覆盖情况：

```
Agent({
  subagent_type: "qa",
  prompt: "作为 QA 专家，分析测试用例与实际代码的覆盖情况，并更新用例状态。\n\n【代码变更记录】\n{CHANGELOG-impl.md 全文}\n\n【现有测试用例】\n{test/case/cases.md 全文（如果存在）}\n\n【后端架构】\n{design/backend/architecture.md 摘要}\n\n【后端 API】\n{design/backend/api.md 摘要}\n\n【前端组件】\n{design/frontend/components.md 摘要}\n\n【测试报告】\n{test/test-report.md 摘要（如果存在）}\n\n【代码审查报告】\n{review/code-review-report.md 摘要（如果存在）}\n\n要求：\n\n1. **覆盖率分析**：\n   - 统计本次代码变更涉及多少 Service/Mapper/Controller/组件\n   - 其中有多少有对应测试用例\n   - 计算覆盖率百分比\n\n2. **测试用例状态更新**：\n   对 cases.md 中的每个测试用例，判断：\n   - ✅ 已实现且通过\n   - ⚠️ 已实现但有问题（测试失败/部分通过）\n   - ❌ 未实现（代码中无对应测试）\n   - 🔄 实现已变更（原用例不适用）\n\n3. **新增测试用例**：\n   对代码中有但测试用例中无覆盖的重要逻辑，补充测试用例说明\n\n4. **覆盖率门禁**：\n   - 核心业务逻辑（Controller/Service）覆盖率 >= 80%\n   - 工具类/Util 覆盖率 >= 60%\n   - 不要求 100% 但需要说明未覆盖的理由\n\n输出格式：\n\n## 测试覆盖率统计\n| 类别 | 变更数 | 已覆盖数 | 覆盖率 |\n|------|--------|---------|--------|\n| 后端 Controller | n | n | nn% |\n| 后端 Service | n | n | nn% |\n| 后端 Mapper | n | n | nn% |\n| 前端组件 | n | n | nn% |\n| 工具类 | n | n | nn% |\n\n## 测试用例状态更新\n| 用例编号 | 原状态 | 新状态 | 说明 |\n|---------|--------|--------|------|\n\n## 新增测试用例建议\n| 用例编号 | 测试目标 | 测试类型 | 优先级 |\n|---------|---------|---------|--------|\n\n## 覆盖率门禁\n| 门禁项 | 要求 | 实际 | 状态 |\n|--------|------|------|------|\n"
})
```

### 4.2 更新测试用例文档

如果 `test/case/cases.md` 存在：
- 更新每个用例的状态列
- 追加新增的测试用例
- 标注未实现或已变更的用例

如果 `test/case/cases.md` 不存在：
- 询问是否需要基于现有代码生成测试用例文档
- 如果需要，启动 Agent 生成完整的测试用例文档
