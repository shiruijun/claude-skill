帮助客户成功部（CSM）将零散的客户反馈转化为结构化、可评估、可排期的敏捷迭代需求文档。

## 状态说明

本 Skill 为**单会话流程**，无持久化状态文件。当前步骤通过对话上下文维护。

- 首次触发：从 Step 1 开始
- 中断后恢复：根据已收集信息判断当前应执行的 Step
- CSM 中途退出：不保存状态，下次重新触发后从头开始

## 执行入口

**首次触发：**

读取 `step-01-greeting.md`，输出欢迎语，进入 Step 2。

**已有部分信息时：**

根据已收集字段判断缺失步骤，从最早未完成的 Step 继续。

## 中断恢复

根据已收集信息判断当前进度：

| 已收集信息 | 从哪继续 |
|-----------|---------|
| 无 | Step 1 |
| 客户信息不完整 | Step 2 |
| 缺少场景三要素 | Step 3 |
| 缺少痛点 | Step 4 |
| 缺少期望方案 | Step 5 |
| 缺少汇总确认（用户故事/功能模块/补充信息） | Step 5.9 |
| 已收集全部信息 | Step 7（AI 自动执行 7~10） |

## 质量门禁

| 门禁 | 要求 | 阻塞 |
|------|------|------|
| 信息完整性 | ≥ 70 分达标，60~69 分合格 | **是** — 低于 60 分禁止生成文档 |
| 场景三要素 | 缺 2 个及以上 | **是** — 强制补充 |
| 客户名称关联 | 未匹配 | 否 — 写入来源备注兜底 |
| Base 写入 | 失败 | 否 — 输出 Markdown 供手动复制 |
| 文档创建 | 失败 | 否 — 输出 Markdown 供手动复制 |

## 注意事项

1. **CSM 是信息传递者：** 不需要具备产品专业知识，只需把客户的话准确、完整地记录下来
2. **不强求完美：** 允许分多次补充
3. **保留原话：** 客户提到的具体描述、竞品名称、数字等，尽量原样记录
4. **不做技术假设：** 不替研发做技术方案判断，只根据业务描述评估规模
5. **规模评估是估算：** 明确告知 CSM 这是初步估算，最终需研发团队评审确认
6. **RICE 是参考：** 分数用于优先级排序参考，不是绝对标准
7. **完整性检查是辅助 AI：** CSM 只需根据提示补充缺失内容，不需要自己判断
8. **飞书授权：** 首次使用需运行 `lark-cli auth login --domain base`
9. **Base 写入失败不阻断：** 仍输出 Markdown 文档，提示手动复制粘贴
10. **附件上传独立执行：** 上传失败不影响记录创建

## 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| CSM 直接粘贴一段客户反馈 | 自动提取信息，对缺失要素发起追问 |
| CSM 说"客户就是想加个小功能" | 追问具体是什么功能、用在哪、解决什么问题 |
| CSM 不清楚影响范围 | 标记为 "仅1家"，备注中说明 "影响范围待确认" |
| 需求明显是 Bug | 标注需求类型为 Bug，规模评估简化 |
| 需求明显是数据需求 | 标注需求类型为数据需求，规模评估侧重数据量和来源 |
| CSM 中途退出 | 不保存状态，下次重新触发后从头开始 |
