# Step 5.5: 用户故事生成（AI 自动）

## 策略

基于已收集的**场景描述**、**痛点描述**、**期望方案**，AI 自动生成标准格式的用户故事。**不需要 CSM 编写，直接展示结果。**

## 生成规则

从已收集的信息中提取三要素：

| 用户故事要素 | 信息来源 | 提取方式 |
|-----------|---------|---------|
| **As a（角色）** | Step 3 场景描述中的角色 | 直接提取 |
| **I want（需求）** | Step 5 期望方案中的期望效果 | 提炼为动作 + 目标 |
| **So that（价值）** | Step 4 痛点描述中的影响 | 转化为"解决后能获得的价值" |

## 生成示例

**输入信息：**
- 角色：财务总监
- 场景：在报销审批页面提交单据
- 痛点：提交后没有成功提示，需要反复刷新确认是否提交成功
- 期望：提交后有明确的提交成功提示，并显示审批进度

**自动生成：**
```
As a 财务总监，
I want 在报销审批提交后看到明确的提交成功提示和审批进度，
So that 我不需要反复刷新确认提交状态，节省操作时间并减少焦虑。
```

## 展示与确认

自动生成后展示给 CSM：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📝 用户故事（AI 自动生成）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

As a {角色}，
I want {期望效果提炼}，
So that {痛点解决后的价值}。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**确认交互：**

> 以上是 AI 根据你提供的信息自动生成的用户故事，产品部将基于这个用户故事理解需求。
>
> 请确认是否准确：
> 1. **准确** → 继续下一步
> 2. **需要修改** → 告诉我哪里不对，我会调整
> 3. **重新生成** → 我会基于更新后的信息重新生成

**修改规则：**
- CSM 可以修改角色、需求描述、价值描述中的任意部分
- AI 根据修改后的内容重新生成完整的用户故事
- 最多允许修改 3 次，超过则记录"用户故事待产品部确认"
