# Step 2: 根因分析（Root Cause Analysis — 强制）

> **策略**：使用 `superpowers:systematic-debugging` 的 Phase 1-3，系统化地找到根因。没有根因分析，不允许提出修复方案。
> 
> **铁律：症状修复即失败。**
> 
> **这是强制阶段，不可跳过。**

## 检查

如果 `phases.rootCauseAnalysis == true` 且 `$BASE_DIR/analysis/root-cause-analysis.md` 存在：

AskUserQuestion（复用/重新执行逻辑同上）。

## 执行

**调用 `systematic-debugging` skill 的完整流程：**

### Phase 1: Root Cause Investigation（根因调查）

#### 1.1 仔细阅读错误信息

```bash
# 收集相关日志
git log --oneline -20 -- {相关文件}

# 查看最近的变更
git diff HEAD~5 HEAD -- {相关文件}
```

- 不要跳过错误或警告
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误代码

#### 1.2 稳定复现

基于 Step 1 的复现报告：
- 能否可靠触发？
- 具体步骤是什么？
- 是否每次都发生？
- 如果不可复现 → 收集更多数据，不要猜测

#### 1.3 检查近期变更

```bash
# 查看相关文件的最近提交
git log --oneline -- {相关文件}

# 查看最近的变更
git show {commit-hash} -- {相关文件}

# 查看最近一周的变更（针对生产 Bug）
git log --since="1 week ago" --oneline -- {相关模块}
```

- 什么变更可能导致此问题？
- 新的依赖、配置变更
- 环境差异

#### 1.4 多组件系统证据收集

**当系统有多个组件时（API → Service → Database）：**

对每个组件边界：
- 记录什么数据进入组件
- 记录什么数据离开组件
- 验证环境/配置传播
- 检查每层的状态

```java
// 示例：在关键位置添加诊断日志
log.info("=== Layer X Debug ===");
log.info("Input: {}", input);
log.info("Output: {}", output);
log.info("Config: {}", configValue);
```

#### 1.5 追踪数据流

当错误深层嵌套在调用栈中时：
- 坏值起源于哪里？
- 什么调用者传入了坏值？
- 持续向上追溯直到找到源头
- **在源头修复，不在症状处修复**

### Phase 2: Pattern Analysis（模式分析）

#### 2.1 查找正常工作的例子

在代码库中定位类似但正常工作的代码：
- 什么工作正常且与出问题的地方类似？
- 使用 `codegraph_search` 查找相似实现

#### 2.2 对比参考实现

- 完整阅读参考实现（不要略读）
- 理解模式的每一个细节

#### 2.3 识别差异

- 正常工作与出问题的代码之间有什么不同？
- 列出每一个差异，无论多小
- 不要假设"那个不可能有影响"

#### 2.4 理解依赖

- 需要什么其他组件？
- 需要什么设置、配置、环境？
- 有什么假设前提？

### Phase 3: Hypothesis and Testing（假设与验证）

#### 3.1 形成单一假设

清晰地陈述："我认为 X 是根因，因为 Y"
- 写下来
- 具体而非模糊

#### 3.2 最小化测试

- 做最小的可能变更来验证假设
- 一次只改变一个变量
- 不要同时修复多个问题

#### 3.3 验证后再继续

- 验证通过？→ 进入 Step 3 修复方案设计
- 验证失败？→ 形成新假设
- **不要在此基础上添加更多修复**

#### 3.4 当无法理解时

- 说"我不理解 X"
- 不要假装知道
- 寻求帮助
- 做更多研究

**如果 3 次以上假设验证失败：**

STOP —— 可能是架构级问题，需要与团队讨论。

## 根因分析文档结构

```markdown
## 根因分析

### 现象描述
{Bug 的具体现象}

### 调查过程
1. {调查步骤 1}
2. {调查步骤 2}
3. {调查步骤 3}

### 根因定位
- **根本原因**: {根因}
- **引入原因**: {如何引入的}
- **影响范围**: {影响的模块/数据}

### 假设验证
| 假设 | 验证方法 | 结果 |
|------|---------|------|
| 假设 1 | {方法} | {通过/失败} |
| 假设 2 | {方法} | {通过/失败} |

### 相关代码
{关键代码片段}

### 根因类别
- [ ] 逻辑错误
- [ ] 边界条件
- [ ] 并发/时序
- [ ] 数据问题
- [ ] 配置错误
- [ ] 第三方库
- [ ] 环境差异
- [ ] 回归问题（近期变更引入）
- [ ] 架构设计缺陷
```
