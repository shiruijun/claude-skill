# Step 3: 修复方案设计（Fix Design — 强制）

> **策略**：基于根因分析，设计最小、最安全的修复方案。不引入新功能，只做必要修改。
> 
> **这是强制阶段，不可跳过。**

## 检查

如果 `phases.fixDesign == true` 且 `$BASE_DIR/analysis/fix-design.md` 存在：

AskUserQuestion（复用/重新执行逻辑同上）。

## 执行

### 3.1 修复方案设计

基于 Step 2 的根因分析，设计修复方案：

```markdown
## 修复方案

### 根因摘要
{引用 root-cause-analysis.md 的根因}

### 修复方案
#### 方案描述
{具体如何修复}

#### 修改范围
| 文件 | 修改类型 | 说明 |
|------|---------|------|
| {文件 1} | 修改/新增/删除 | {说明} |
| {文件 2} | 修改/新增/删除 | {说明} |

#### 修复代码（草稿）
```java
// 修复前的代码
{原代码}

// 修复后的代码
{修复后的代码}
```

### 影响分析
- **直接影响**: {修复直接解决的问题}
- **潜在影响**: {可能影响的周边功能}
- **数据影响**: {是否需要数据修复}
- **兼容性**: {是否影响 API 兼容性}

### 回滚方案
{如果修复引入新问题，如何回滚}

### 数据修复（如需要）
```sql
-- 数据修复脚本
```
```

### 3.2 方案评审检查清单

- [ ] 修复是否在根因处，而非症状处？
- [ ] 修改范围是否最小化？
- [ ] 是否避免了"顺便优化"？
- [ ] 是否考虑了边界条件？
- [ ] 是否考虑了并发/时序问题？
- [ ] 是否需要数据迁移/修复？
- [ ] 是否影响现有 API 兼容性？
- [ ] 是否有回滚方案？
- [ ] 是否需要更新文档？

### 3.3 评审确认

如果修复涉及架构变更或高风险修改：

AskUserQuestion：
> 修复方案设计完成，请确认：
> - **A)** 确认，进入 TDD 修复阶段
> - **B)** 需要调整方案
> - **C)** 方案有架构风险，需要进一步讨论

## 产出

使用 [templates/fix-design.md](templates/fix-design.md) 模板，写入 `$BASE_DIR/analysis/fix-design.md`。

## 确认

Read 工具读取 `fix-design.md`，AskUserQuestion：
> 修复方案确认？
> - **A)** 确认，进入 TDD 修复阶段
> - **B)** 需要调整方案
> - **C)** 重新设计

确认后更新 `phases.fixDesign = true`。
